This specification defines a Data Integrity cryptosuite for expressing [[ANONCREDS]] credentials and presentations as [[VC-DATA-MODEL]] documents. AnonCreds CL signatures and zero-knowledge proofs are carried in DataIntegrityProof envelopes identified by the anoncreds-2023 cryptosuite string. CL cryptographic operations remain defined by [[ANONCREDS]]; this document specifies the W3C data model, verificationMethod binding to credential definitions, and proofValue encoding.

This is an editor's draft. Implementations SHOULD track anoncreds-rs and the examples in examples/.

Comments and pull requests are welcome via GitHub.

Introduction

[[ANONCREDS]] defines Camenisch–Lysyanskaya (CL) credential issuance, selective-disclosure presentations, and verification. [[VC-DATA-INTEGRITY]] defines a modern proof envelope (DataIntegrityProof) and requirements for cryptosuite specifications such as vc-di-bbs.

This document bridges the two: it normatively describes how AnonCreds proofs appear in W3C verifiable credentials and presentations. The reference encoding is implemented in anoncreds-rs ([[ANONCREDS-RS]]).

Relation to legacy AnonCreds W3C mapping

The [[ANONCREDS]] document also describes AnonCredsProof2023 and AnonCredsPresentationProof2023 proof types. Those legacy envelopes remain valid for existing deployments. New implementations SHOULD use anoncreds-2023 as defined here.

Cryptosuite identifier

This specification defines one cryptosuite string: anoncreds-2023. The kind of AnonCreds proof (credential signature, per-credential presentation sub-proof, or aggregated presentation proof) is determined by the decoded proofValue (see ), not by separate cryptosuite identifiers.

Conformance

As well as sections marked as informative, all examples and notes in this specification are non-normative. Everything else is normative.

The key words MAY, MUST, MUST NOT, OPTIONAL, REQUIRED, and SHOULD in this document are to be interpreted as described in [[RFC2119]] [[RFC8174]] when, and only when, they appear in all capitals.

A conforming proof complies with the normative statements in Data Model.

A conforming processor generates or verifies conforming proofs and MUST error when required properties are missing or proofValue cannot be decoded.

Data Model

The following sections define verification methods and proof formats for the anoncreds-2023 cryptosuite.

Verification Methods

In AnonCreds, the public verification material for a credential is the credential definition published on a [[ANONCREDS]] verifiable data registry. Unlike eddsa-jcs-2022 or bbs-2023, the issuer's CL public key is not typically expressed as a Multikey on a DID document; it is resolved from the credential definition object.

verificationMethod identifier

For proofs using the anoncreds-2023 cryptosuite, the verificationMethod property of a DataIntegrityProof MUST be a [[URL]] whose value is the credential definition identifier (cred_def_id).

The same identifier MUST appear inside the decoded proofValue as credDefId (MessagePack field cred_def_id) for credential and per-credential presentation proofs.

Example 1verificationMethod on a credential proof
"verificationMethod": "creddef:government"

Resolution

Dereferencing verificationMethod MUST yield the published credential definition for that identifier using the applicable [[ANONCREDS]] AnonCreds Method on the target verifiable data registry. The CL public key material in that object is used to verify proofs, as described in [[ANONCREDS]] issuance and presentation sections.

This specification does not define a new JSON-LD verification method type (such as Multikey). Processors resolve verificationMethod as a credential definition identifier and obtain CL keys from the registry object. A future revision may define an explicit AnonCredsCredentialDefinition verification method type for DID-based publication.

Verifiable Credential and Presentation Documents

AnonCreds credentials and presentations are [[VC-DATA-MODEL]] JSON-LD documents with the following conventions (aligned with anoncreds-rs).

@context

Documents using the anoncreds-2023 cryptosuite are [[VC-DATA-MODEL]] (or [[VC-DATA-MODEL-2.0]]) documents. The @context array MUST satisfy the requirements of the applicable verifiable credentials data model version, including that every term used in the document is defined through one or more JSON-LD contexts as described in those specifications.

In particular, implementers MUST ensure that DataIntegrityProof and its properties (cryptosuite, proofPurpose, proofValue, verificationMethod, and challenge where used) are mapped to IRIs through @context in a way that is safe for their processing environment. Conforming processors that perform JSON-LD processing SHOULD run context validation as described in [[VC-DATA-INTEGRITY]] or apply equivalent protections.

This specification does not define a dedicated AnonCreds JSON-LD context and does not prescribe a fixed @context list. Issuers and wallets choose contexts that define credential vocabulary (for example issuer-dependent terms for credentialSubject properties) alongside the base verifiable credentials context for the target data model version.

The anoncreds-rs reference implementation, when generating documents, uses the contexts shown in examples/sample_credential.json. That choice is illustrative; other valid context arrangements are permitted if they meet the requirements above.

type

A verifiable credential MUST include VerifiableCredential in type. A verifiable presentation MUST include VerifiablePresentation. This specification does not require an additional AnonCredsCredential type string; AnonCreds semantics are conveyed by the anoncreds-2023 proof.

Credential body

The credential document MUST include issuer and credentialSubject. Schema and credential-definition identifiers are carried inside proofValue, not in a top-level credentialSchema property.

Presentation body

A presentation MUST include verifiableCredential as an array of credential documents. Each embedded credential carries its own DataIntegrityProof (per-credential sub-proof). The presentation MUST include a top-level proof with the aggregated presentation proof.

Proof Representations

DataIntegrityProof

Proofs MUST use the DataIntegrityProof type defined in [[VC-DATA-INTEGRITY]]. The following constraints apply to the anoncreds-2023 cryptosuite:

PropertyRequirement
type MUST be DataIntegrityProof.
cryptosuite MUST be anoncreds-2023.
verificationMethod MUST be the credential definition id (see ).
proofPurpose MUST be assertionMethod for credential proofs and per-credential proofs inside a presentation; MUST be authentication for the top-level presentation proof.
proofValue MUST be present; see .
challenge MUST be present on the top-level presentation proof and MUST equal the presentation-request nonce from [[ANONCREDS]]. Absent on credential proofs.
Example 2Credential proof envelope
{
  "type": "DataIntegrityProof",
  "cryptosuite": "anoncreds-2023",
  "proofPurpose": "assertionMethod",
  "verificationMethod": "creddef:government",
  "proofValue": "ukgGEqXNjaGVtYV9pZLFzY2hlbWE6Z292ZXJubWVudK..."
}
Example 3Presentation proof envelope
{
  "type": "DataIntegrityProof",
  "cryptosuite": "anoncreds-2023",
  "proofPurpose": "authentication",
  "verificationMethod": "creddef:government",
  "challenge": "191952109004431815119609",
  "proofValue": "ukgOBqmFnZ3JlZ2F0ZWSCpmNfaGFzaNwAI..."
}

proofValue

The proofValue property carries the AnonCreds cryptographic material. It MUST be a string using multibase-base64url-no-pad encoding per [[CID-1.0]]: a leading u (U+0075) followed by the base64url-no-pad encoding of a binary blob.

The binary blob MUST be a [[MSGPACK]] document (using to_vec_named / map field names as defined below) representing a two-element array [tag, payload].

Proof tags
TagNameUsed on
1 Credential signature proof Issued verifiable credential (before or after holder processing)
2 Credential presentation proof Each verifiableCredential entry inside a presentation
3 Presentation proof Top-level presentation proof

Any other tag value MUST be rejected.

Tag 1 — Credential signature payload

The payload object MUST contain:

FieldRequiredDescription
schema_idYesAnonCreds schema identifier
cred_def_idYesCredential definition identifier; same as verificationMethod
rev_reg_idNoRevocation registry definition id when credential is revocable
signatureYesCL credential signature per [[ANONCREDS]]
signature_correctness_proofYesSignature correctness proof per [[ANONCREDS]]
rev_regNoRevocation registry accumulator snapshot at issuance
witnessNoWitness for revocation registry at issuance
Tag 2 — Credential presentation payload

The payload object MUST contain:

FieldRequiredDescription
schema_idYesSchema identifier for the source credential
cred_def_idYesCredential definition identifier
rev_reg_idNoRevocation registry when non-revocation proof is included
timestampNoUnix timestamp for revocation registry state used in NRP
sub_proofYesCL sub-proof (primary_proof, optional non_revoc_proof) per [[ANONCREDS]]
Tag 3 — Presentation payload

The payload object MUST contain:

FieldRequiredDescription
aggregated Yes Aggregated CL proof (c_hash, c_list) per [[ANONCREDS]]
Decoding algorithm (informative)
  1. If proofValue does not start with u, fail.
  2. Base64url-no-pad-decode the remainder to bytes.
  3. MessagePack-decode bytes to array [tag, payload].
  4. If tag is not 1, 2, or 3, fail.
  5. Validate payload fields per tag section above.

Multiple proofs on credentials

A verifiable credential proof MAY be an array. Exactly one entry MUST use the anoncreds-2023 cryptosuite. Additional DataIntegrityProof entries using other cryptosuites (for example eddsa-jcs-2022) MAY be present; verification of non-AnonCreds proofs is outside the scope of this specification.
